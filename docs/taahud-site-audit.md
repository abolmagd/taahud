# تقرير فحص موقع تعاهُد

تاريخ الفحص: 17 يوليو 2026  
النطاق: واجهة الطالب، لوحة الإدارة، منطق الإحصائيات، Supabase، التصميم المتجاوب، والوصول WCAG 2.1 AA.

## ملخص تنفيذي

الرحلات الأساسية تعمل في الاختبارات: دخول الطالب، تغيير الباسورد، تسجيل الجلسة، الإحصائيات والسجلات، وتنقلات لوحة الإدارة. نجحت 34 وحدة اختبار، ونجح فحص Playwright على 1280px و390px بلا أخطاء JavaScript أو overflow أفقي.

لكن لا أنصح بتوسيع الاستخدام قبل إصلاح مشاكل الأمان وسلامة النقاط التالية:

1. الحسابات غير المفعلة فعليًا قابلة للاستيلاء بسبب الباسورد الافتراضي المشترك.
2. الطالب يستطيع اختيار نفسه كسامع والحصول على نقاط الدورين.
3. لوحة الإدارة تعيد حساب التاريخ بالقواعد الحالية، بينما الطالب يرى النقاط المخزنة وقت التسجيل.
4. التسجيل بأيام قديمة مفتوح بلا حد زمني أو مراجعة، ولا يوجد منع للتكرار أو قيم الصفحات غير المنطقية.

## مشاكل الوظائف وتجربة الاستخدام

### واجهة الطالب

- الجلسة لا تستمر بعد تحديث الصفحة؛ الكود والباسورد محفوظان في الذاكرة فقط. الحل الآمن هو session token قصير العمر، وليس حفظ الباسورد في `localStorage`.
- لا توجد طريقة لتغيير الباسورد بعد أول مرة أو استعادته ذاتيًا.
- اختيار `وِرد استماع` لا يغيّر الأسئلة غير المناسبة مثل جودة المحفوظ وطريقة التسميع.
- رسالة الخطأ غالبًا عامة ولا تحدد الحقل الخاطئ، والنموذج يستخدم `novalidate` مع Toast واحد.
- بعد نجاح التسجيل يتم مسح النموذج، لكن لا يظهر ملخص دائم أو زر مباشر لفتح السجل الجديد.
- لا يوجد تأكيد للطرف الآخر أو حماية من اختيار كود خاطئ.

### لوحة الإدارة

- لا توجد loading states أو وقت "آخر تحديث"، لذلك يصعب التمييز بين صفر حقيقي وبيانات لم تصل بعد.
- سجل الجلسات يفلتر بالطالب والطريقة فقط؛ ينقصه التاريخ، الدور، نوع السامع، والبحث النصي.
- كل الجلسات تُحمّل كاملة في كل refresh، مع Realtime وpolling كل 15 ثانية. هذا سيتباطأ ويزيد استهلاك Supabase مع نمو البيانات.
- لا توجد pagination أو server-side filtering أو export CSV.
- إيقاف الطالب لا يطلب تأكيدًا، ولا يوجد سجل عمليات للإضافة والإيقاف والـreset والإعدادات.
- لا توجد إضافة جماعية للطلاب أو توليد PINs فردية وطباعتها/تصديرها.
- لا توجد empty state واضحة في جدول الإحصائيات وسجل الإدارة عندما تكون النتيجة صفرًا.

## الوصول WCAG 2.1 AA

الفحص الآلي لواجهة الطالب لم يجد مشاكل HTML ثابتة. لوحة الإدارة لديها تحذيران Level A:

- `log-filter-student` و`log-filter-method` بلا labels أو `aria-label`.

كما لا يوجد `<main>` landmark في لوحة الإدارة، وهو تحسين بنيوي إضافي أوصى به الفحص.

والفحص الديناميكي وجد:

- 6 عناوين أعمدة قابلة للفرز بالماوس فقط وليست قابلة للتركيز بلوحة المفاتيح.
- Toasts الطالب والإدارة بلا `aria-live` أو `role=status/alert`، فلا يعلنها قارئ الشاشة.
- حدود الحقول `#ddd6c7` على الأبيض تعطي contrast يساوي 1.45:1، أقل من 3:1 المطلوبة للمكونات المرئية.
- تبديل الصفحات والتبويبات لا ينقل focus إلى عنوان المحتوى الجديد.

## التصميم

### ما تم إصلاحه الآن

- Chart آخر 7 أيام أصبح بعرض لوحة الإدارة كاملة.
- ارتفاع العمود يعكس النقاط فعليًا، وأيام الصفر لا تعرض أعمدة وهمية.
- كل يوم يعرض النقاط، اسم اليوم، التاريخ، عدد الجلسات، والصفحات.
- على الموبايل يتحول الرسم إلى صفوف أفقية واضحة بدل أعمدة مضغوطة.
- الرسم له وصف نصي كامل لقارئات الشاشة.

### تحسينات مقترحة

- تقليل ارتفاع الهيدر بعد الدخول في الطالب والإدارة لإظهار المحتوى الأساسي أسرع على الموبايل.
- تحويل tabs الإدارة على الهاتف إلى شريط أفقي قابل للسحب أو تنقل سفلي ثابت بدل الالتفاف غير المنتظم.
- تحويل جداول الإدارة على الموبايل إلى rows تفصيلية مثل سجلات الطالب، مع إبقاء الجدول للديسكتوب.
- إضافة حالات loading skeleton وerror state وempty state لكل لوحة، لا عرض الصفر فقط.
- توحيد الاسم إلى `السجلات` في الطالب والإدارة.
- إضافة أيقونات للأوامر المتكررة مثل خروج، رجوع، تصدير، تعديل، وحذف مع tooltips.
- جعل الحقول والحواف أوضح بصريًا ورفع contrast الحدود.
- في نموذج الطالب، تغيير النصوص والحقول حسب نوع الجلسة لتقليل الالتباس.

## إحصائيات تستحق الإضافة

1. مقارنة الفترة الحالية بالفترة السابقة: الجلسات، الصفحات، النقاط، والطلاب النشطون مع نسبة التغير.
2. توزيع الستريك: 0، 1-2، 3-6، 7+ أيام.
3. الطلاب المعرضون للانقطاع: لم يسجلوا اليوم أو منذ 3/7 أيام.
4. معدل العودة الأسبوعي وعدد الطلاب النشطين يوميًا وأسبوعيًا.
5. متوسط ووسيط الصفحات، لأن المتوسط وحده يتأثر بالقيم الكبيرة.
6. نسبة الجلسات داخل تعاهُد مقابل خارجها وورد الاستماع عبر الزمن.
7. تقرير جودة البيانات: self-listener، تكرار، صفحات غير منطقية، تسجيلات قديمة، وقيم غير معروفة.
8. Drill-down من أي رقم أو bar إلى الجلسات التي كوّنته.
9. تصدير CSV/PDF للفترة المحددة.
10. تعريف صغير لكل metric يوضح هل النقاط تاريخية أم محسوبة بالقواعد الحالية.

## خطة تنفيذ مقترحة

### المرحلة 1: قبل التوسع

1. استبدال الباسورد المشترك بـPIN فردي وتأمين قائمة الطلاب.
2. منع self-listener وإضافة rate limiting.
3. حسم سياسة النقاط التاريخية وتوحيد الطالب والإدارة.
4. وضع حدود للتاريخ والصفحات ومنع التكرار.

### المرحلة 2: سلامة التشغيل

1. تعديل/إلغاء الجلسات مع audit log.
2. pagination وserver-side filtering وCSV.
3. labels، live regions، keyboard sorting، contrast، وfocus management.
4. loading/error/empty states وآخر وقت تحديث.

### المرحلة 3: التطوير والتحليلات

1. جلسات دخول آمنة للطلاب واستعادة/تغيير الباسورد.
2. مقارنة الفترات، retention، توزيع الستريك، وتنبيهات الانقطاع.
3. تحسين الموبايل والجداول والتنقل.

## التحقق المنفذ

- 34/34 وحدة اختبار ناجحة.
- فحص JavaScript syntax ناجح لكل الملفات الرئيسية.
- Playwright: واجهة الطالب ولوحة الإدارة على 1280x900 و390x844 بلا page errors أو viewport overflow.
- اختبار رحلة الطالب: دخول، تنقل بين التسجيل والإحصائيات والسجلات، وإرسال جلسة.
- اختبار لوحة الإدارة: overview، الطلاب، الإعدادات، الإحصائيات، والسجل.
- فحص WCAG ثابت وديناميكي وفحص contrast للألوان الأساسية.
- فحص REST الحي أكد أن جلسات الطلاب نفسها غير قابلة للقراءة للـanon، لكن بيانات الطلاب والإعدادات قابلة للقراءة.

## حدود الفحص

لم أستخدم باسورد الإدارة الحقيقي، لذلك عمليات الكتابة الحية مثل إضافة طالب وحفظ الإعدادات وreset لم تُنفذ على بيانات الإنتاج. تم اختبار مساراتها بمزود Supabase وهمي ومراجعة الكود والسياسات مباشرة لتجنب تغيير بيانات حقيقية أثناء الفحص.
